# Инструкция для ИИ-агента: рефакторинг DrumMachine

## Роль

Ты работаешь как ведущий Godot-разработчик. Твоя задача — улучшить архитектуру,
читаемость, расширяемость и тестируемость проекта DrumMachine без изменения
существующего пользовательского поведения.

Проект использует Godot 4.6 и GDScript.

## Главная цель

Постепенно привести код к структуре, в которой:

- `main.gd` является небольшим координатором сцены;
- состояние паттерна можно целиком сохранять, восстанавливать и передавать в
  историю undo/redo;
- UI-компоненты отвечают только за отображение и пользовательский ввод;
- модель содержит бизнес-правила и проверку данных;
- модули общаются через явные методы и сигналы;
- новые функции вроде solo, pan, индивидуальной громкости и новых параметров
  дорожки можно добавлять без разрастания `main.gd`;
- основная логика покрыта автоматическими тестами.

## Обязательные ограничения

1. Сначала изучи текущую реализацию и существующие изменения в Git.
2. Не отменяй изменения пользователя и не редактируй несвязанные файлы.
3. Выполняй рефакторинг небольшими законченными этапами.
4. После каждого этапа проект должен запускаться.
5. Не меняй внешний вид, управление и текущее поведение без явной причины.
6. Сохрани поддержку:
   - короткого нажатия по пэду;
   - долгого нажатия или долгого тапа;
   - правой кнопки мыши для открытия velocity popup;
   - touch-ввода;
   - воспроизведения, mute, выбора звука, слотов и горячих клавиш.
7. Не добавляй новый `Manager`, если ответственность можно выразить отдельным
   UI-компонентом, моделью, сервисом или сигналом.
8. Не выполняй полный rewrite проекта.
9. Используй строгую типизацию GDScript там, где она улучшает контракты.
10. Не оставляй временные заглушки, мёртвый код и неиспользуемые API.

## Целевая архитектура

Ориентируйся на следующие роли. Названия можно адаптировать к существующему
стилю проекта.

### `Sequencer`

Чистая модель приложения:

- сетка шагов;
- дорожки;
- tempo и количество шагов;
- mute и пути к звукам;
- ограничения и проверка значений;
- операции изменения паттерна;
- создание и восстановление полного снимка состояния.

Модель не должна обращаться к UI, `Timer`, popup или `AudioStreamPlayer`.

### `PatternState`

Полный снимок редактируемого состояния:

- версия формата;
- rows;
- steps;
- tempo;
- grid;
- muted;
- sounds.

Состояние должно безопасно копироваться, сериализоваться и валидироваться.
Можно реализовать его как `Resource`, `RefCounted` или строго определённый
словарь, но контракт должен быть централизован.

### `PatternHistory`

История должна хранить полный `PatternState`, а не только grid. Undo/redo после
загрузки, изменения числа шагов, добавления или удаления строки должен
восстанавливать всё состояние согласованно.

### `TransportController`

Ответственность:

- play/stop;
- управление таймером;
- изменение tempo;
- tap tempo;
- сброс позиции воспроизведения.

Он не должен строить UI строк или управлять popup.

### `DrumRow`

Отдельная сцена и скрипт одной дорожки:

- имя и цвет;
- кнопка mute;
- кнопка выбора звука;
- кнопка удаления;
- отображение состояния строки.

Компонент сообщает о действиях сигналами, например:

```gdscript
signal mute_changed(row: int, muted: bool)
signal sound_requested(row: int, anchor: Control)
signal delete_requested(row: int)
```

Компонент не должен напрямую изменять `Sequencer`.

### Компонент взаимодействия с пэдами

Логика короткого нажатия, долгого нажатия, touch и правой кнопки должна
находиться в одном месте и иметь понятное состояние. Не дублируй обработку
одного жеста между `main.gd` и `StepGridBuilder`.

### `PopupManager`

Допустимо сохранить единый координатор popup, но:

- popup должны иметь собственные сигналы закрытия;
- не используй хрупкие строковые пути `get_node()` к внутренностям popup;
- повторяющуюся анимацию открытия вынеси в общий приватный метод;
- публичные ссылки на popup замени методами вроде `has_velocity_popup()`.

### Сохранение

`SaveManager` отвечает только за чтение и запись данных. Он должен:

- проверять номер слота;
- возвращать результат операции или понятную ошибку;
- поддерживать версию формата;
- безопасно обрабатывать отсутствующий, повреждённый или несовместимый JSON.

Координация UI сохранения и загрузки должна находиться отдельно от файлового
доступа.

## План выполнения

### Этап 1. Зафиксировать текущее поведение

1. Запусти проект и проверь отсутствие ошибок GDScript.
2. Составь краткую карту зависимостей.
3. Добавь тесты для текущей чистой логики:
   - создание `Sequencer`;
   - изменение velocity;
   - resize steps с сохранением данных;
   - add/remove row;
   - clear и randomize;
   - сериализация и восстановление;
   - undo/redo.
4. Не начинай структурный рефакторинг до появления базовой проверки модели.

### Этап 2. Полное состояние паттерна

1. Введи единый `PatternState`.
2. Добавь версию формата сохранения.
3. Централизуй копирование и восстановление состояния.
4. Ограничивай входные данные:
   - `tempo`: `Sequencer.MIN_TEMPO..Sequencer.MAX_TEMPO`;
   - `steps`: `Sequencer.MIN_STEPS..Sequencer.MAX_STEPS`;
   - `rows`: `Sequencer.MIN_ROWS..Sequencer.MAX_ROWS`;
   - velocity: `0..4`.
5. Проверяй тип каждой строки grid и каждого элемента массива.
6. Обнови `PatternHistory`, чтобы он хранил полные снимки.
7. Проверь undo/redo после:
   - загрузки слота;
   - изменения количества шагов;
   - добавления строки;
   - удаления строки;
   - изменения mute и звука, если эти операции включены в историю.

### Этап 3. Инкапсуляция `Sequencer`

Убери прямые изменения публичных полей из контроллеров. Добавь методы с
валидацией:

```gdscript
func set_tempo(value: int) -> void
func change_tempo(delta: int) -> void
func set_muted(row: int, muted: bool) -> void
func set_sound(row: int, path: String) -> void
func set_velocity(row: int, step: int, velocity: int) -> void
```

Добавь только действительно полезные сигналы:

```gdscript
signal tempo_changed(value: int)
signal structure_changed
signal row_changed(row: int)
signal pattern_changed
```

Не создавай сигнал на каждое поле без практической необходимости.

### Этап 4. Компонент строки

1. Создай `drum_row.tscn` и `drum_row.gd`.
2. Перенеси туда создание и настройку кнопок строки из `_create_row_ui()`.
3. Подключай действия через сигналы.
4. Удали параллельные массивы `_row_labels`, `_row_mute_btns`,
   `_row_snd_btns`, если их можно заменить массивом `DrumRow`.
5. Сохрани существующие стили, tooltip, mute-анимацию и удаление строк.

### Этап 5. Взаимодействие с пэдами

1. Объедини обработку mouse и touch в одном компоненте.
2. Явно различай:
   - короткое нажатие;
   - отпускание;
   - отменённое нажатие;
   - long press;
   - правую кнопку мыши.
3. Таймер long press должен быть переиспользуемым, а не создаваться при каждом
   событии.
4. Popup velocity должен открываться одинаково через long press и ПКМ.
5. Исключи случайное изменение velocity после сработавшего long press.

### Этап 6. Транспорт и операции паттерна

1. Вынеси play/stop, tempo и tap tempo в `TransportController`.
2. Убери дублирование pause/rebuild/resume.
3. Введи один безопасный способ выполнить структурное изменение:

```gdscript
func mutate_structure(action: Callable) -> void
```

Либо используй более явные методы без универсального `Callable`, если они
лучше читаются.

4. Вынеси clear, randomize, undo и redo в отдельный контроллер только если это
реально уменьшает связанность. Не дроби код механически.

### Этап 7. Сигналы вместо обратных `Callable`

Замени большие `setup(...)` с множеством callback-параметров на сигналы или
узкие зависимости. В первую очередь переработай `SlotManager`.

Плохо:

```gdscript
setup(owner, seq, popups, music, history, timer, grid,
	on_rebuild, on_apply_sounds, on_refresh_tempo, on_refresh_steps)
```

Предпочтительно:

```gdscript
signal pattern_loaded(state: PatternState)
signal pattern_saved(slot: int)
signal operation_failed(message: String)
```

### Этап 8. Sound manifest

1. Не записывай данные в `res://` во время работы экспортированного приложения.
2. Используй один из вариантов:
   - заранее сгенерированный manifest;
   - editor tool для генерации;
   - runtime-кеш в `user://`.
3. Отдели обнаружение аудиофайлов от UI `SoundPicker`.
4. Сделай порядок категорий и файлов детерминированным.

### Этап 9. Очистка

Проверь и удали либо реализуй:

- пустой `scripts/ui/step_button.gd`;
- пустой `_animate_row_in()`;
- неиспользуемые поля `MuteController`;
- неиспользуемый `MusicManager.get_player()`;
- неиспользуемый `DrumTheme.MAX_VISIBLE_ROWS`;
- незавершённую обёртку `Localization`.

Не сохраняй абстракции «на будущее», если они сейчас ничего не делают.

### Этап 10. Типизация и документация

Используй конкретные типы:

```gdscript
var _row_views: Array[DrumRow] = []
var _slot_buttons: Array[Button] = []
var _buttons: Array[Array] = []
```

Для сложных вложенных коллекций допустим небольшой именованный тип вместо
непонятного `Dictionary`.

Комментарии должны объяснять причину или контракт, а не повторять код.
Обнови `README.md`, добавив:

- структуру проекта;
- управление;
- формат сохранения;
- запуск проверок;
- краткое описание архитектуры.

## Дополнительные правила качества

- Не индексируй `DrumTheme.ROW_COLORS[row]` без безопасного ограничения или
  метода `row_color(row)`.
- Проверяй границы row, step, slot и velocity на публичных входах.
- `MusicManager.play_row()` должен безопасно обрабатывать некорректную velocity
  и отсутствующий stream.
- Методы сохранения должны сообщать об ошибке, а не молча завершаться.
- Не допускай нескольких активных popup, если они конфликтуют.
- При переключении сцены корректно останавливай таймеры и временные операции.
- Не смешивай файловый доступ, построение UI и изменение модели в одном классе.

## Проверки после каждого этапа

Выполняй минимум:

```bash
git diff --check
HOME=/tmp/godot-drum-test godot --headless --path . --editor --quit
HOME=/tmp/godot-drum-test godot --headless --path . --quit-after 3
```

Также запускай добавленные тесты. Если команда не может выполниться из-за
ограничений окружения, явно укажи это в отчёте.

Проводи ручную проверку:

1. Play/stop и изменение tempo.
2. Tap tempo.
3. Изменение количества шагов.
4. Короткое нажатие на пэд.
5. Long press и touch.
6. ПКМ по пэду.
7. Mute.
8. Выбор звука.
9. Add/delete row.
10. Save/load всех слотов.
11. Undo/redo после каждого типа изменения.
12. Keyboard shortcuts.

## Критерии готовности

Работа завершена, когда:

- `main.gd` заметно уменьшен и выполняет роль координатора;
- полный паттерн восстанавливается через undo/redo;
- повреждённые сохранения не приводят к падению;
- UI строк представлен отдельной сценой;
- обработка жестов не дублируется;
- `SlotManager` не зависит от большого набора UI callback;
- runtime не пытается писать в `res://`;
- удалён мёртвый код;
- ключевые коллекции типизированы;
- тесты модели и сериализации проходят;
- Godot загружает проект без ошибок скриптов;
- существующее управление и внешний вид сохранены.

## Формат отчёта агента

После каждого этапа сообщай:

1. Что изменено.
2. Какие файлы затронуты.
3. Какие архитектурные зависимости удалены или добавлены.
4. Какие проверки выполнены и их результат.
5. Какие риски или следующие этапы остались.

Не ограничивайся планом: если этап однозначен и не требует решения
пользователя, реализуй его, проверь и только затем переходи к следующему.
